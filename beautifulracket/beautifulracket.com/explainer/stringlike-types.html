Stringlike types



Racket’s string­like types include strings, sym­bols, iden­ti­fiers, key­words, and path strings. They are “string­like” in the sense that they all con­sist of a sequence of Uni­code char­ac­ters, and can eas­ily be con­verted among one another.



Strings



A stan­dard string is enclosed in "double quotes". A lit­eral dou­ble-quote char­ac­ter can be escaped as \".


"one string"
"two \"strings\""
1
2
"one string"
"two \"strings\""


By default, strings will print as lit­eral expres­sions (mean­ing, their printed form can be reused as input). To print strings in the usual friendly way, use display or displayln. To print strings in the raw style used on the REPL, use print or println:


"one literal string\n"
"two literal \"strings\""
(display "one displayed string\n")
(displayln "two displayed \"strings\"")
(print "one printed string\n")
(println "two printed \"strings\"")
1
2
3
4
5
6
"one literal string\n"
"two literal \"strings\""
(display "one displayed string\n")
(displayln "two displayed \"strings\"")
(print "one printed string\n")
(println "two printed \"strings\"")

"one literal string"
"two literal \"strings\""
one displayed string
two displayed "strings"
"one printed string\n""two printed \"strings\""
1
2
3
4
5
"one literal string"
"two literal \"strings\""
one displayed string
two displayed "strings"
"one printed string\n""two printed \"strings\""


Strings can also include the usual escape char­ac­ters to rep­re­sent ASCII white­space, like \t for tab and \n for line­feed, as well as spe­cial classes of escape char­ac­ters like \x for hex num­bers and \u for Uni­code char­ac­ters. (See the Racket docs for the full list.)



Racket also has the usual string-manip­u­la­tion func­tions—string-split, string-join, string-append, format, etc.—that do their usual thing.

Here strings & @-expres­sions



To make mul­ti­line strings inside a pro­gram, two options.





multiline string
that goes on
seemingly
forever.
here-string-delimiter
) ; contrary to usual Racket style,
  ; the closing paren of a here string must go on a new line

(display str)

Sym­bols



A sym­bol is like a string, but writ­ten with a sin­gle ' pre­fix rather than sur­round­ing dou­ble quotes. A sym­bol can­not con­tain spaces unless the whole sym­bol is sur­rounded with | ver­ti­cal bars |:


"string"
'symbol
"two-word string"
'|two-word symbol|
1
2
3
4
"string"
'symbol
"two-word string"
'|two-word symbol|


Sym­bols have one spe­cial prop­erty that strings do not: they are interned, mean­ing Racket guar­an­tees that all instances of a par­tic­u­lar sym­bol refer to only one object in mem­ory. There­fore, it’s a good habit to use sym­bols when you need high-speed equal­ity check­ing, because sym­bols can be com­pared with eq?, which is much faster than the equal? required by strings (see also equal­ity):


(equal? "string" "string") ; slower
(equal? 'symbol 'symbol) ; slower
(eq? 'symbol 'symbol) ; wicked fast
1
2
3
(equal? "string" "string") ; slower
(equal? 'symbol 'symbol) ; slower
(eq? 'symbol 'symbol) ; wicked fast


string->symbol and symbol->string are avail­able for explicit con­ver­sions.



Iden­ti­fiers



An iden­ti­fier is a name that appears in a pro­gram. For its behav­ior, see iden­ti­fiers. Within a datum, an iden­ti­fier is rep­re­sented with a sym­bol:


(define foo "see")
(define bar "saw")
(string-append foo bar) ; "seesaw"
'(string-append foo bar) ; '(string-append foo bar)
(andmap symbol? '(string-append foo bar)) ; #t
1
2
3
4
5
(define foo "see")
(define bar "saw")
(string-append foo bar) ; "seesaw"
'(string-append foo bar) ; '(string-append foo bar)
(andmap symbol? '(string-append foo bar)) ; #t


Here, (string-append foo bar) is an S-expres­sion con­sist­ing of a list con­tain­ing three iden­ti­fiers. When eval­u­ated nor­mally, it becomes "seesaw". When con­verted to a datum, it becomes a list of three sym­bols (= 'string-append, 'foo, 'bar).



Con­versely, when a datum is eval­u­ated, sym­bols inside are treated as iden­ti­fiers. For instance, when the three sym­bols 'cons, 'foo, and 'bar are grouped into a list and eval­u­ated, they are treated as iden­ti­fiers:


(define foo "see")
(define bar "saw")
(define datum (list 'cons 'foo 'bar))
(evaluate datum) ; '("see" . "saw")
1
2
3
4
(define foo "see")
(define bar "saw")
(define datum (list 'cons 'foo 'bar))
(evaluate datum) ; '("see" . "saw")


In these sit­u­a­tions, if you really want a sym­bol in the result, then put two ' pre­fixes on the orig­i­nal item:


(define foo "see")
(define bar "saw")
(define datum (list 'cons ''foo ''bar))
(evaluate datum) ; '(foo . bar)
1
2
3
4
(define foo "see")
(define bar "saw")
(define datum (list 'cons ''foo ''bar))
(evaluate datum) ; '(foo . bar)


Most strings can be con­verted into iden­ti­fiers, though there are some reserved char­ac­ters and spe­cial con­ven­tions. For instance, strings rep­re­sent­ing num­bers can­not be used as iden­ti­fiers.



Key­words



A key­word is used to intro­duce a key­word argu­ment either in a func­tion def­i­n­i­tion, or a call to that func­tion. Below, #:base and #:pow are key­words:


(define (f #:base x #:pow y)
  (expt x y))
(f #:base 2 #:pow 8) ; 256
1
2
3
(define (f #:base x #:pow y)
  (expt x y))
(f #:base 2 #:pow 8) ; 256


It’s usu­ally only nec­es­sary to think about key­words as data if you’re writ­ing macros or other spe­cial oper­a­tions that pro­gram­mat­i­cally manip­u­late key­word argu­ments, in which case you can use string->keyword and keyword->string to con­vert.


(define (f #:base x #:pow y)
  (expt x y))
(f #:base 2 #:pow 8) ; 256
(keyword-apply f '(#:base #:pow) '(2 8) empty) ; 256
(keyword-apply f (list
                  (string->keyword "base")
                  (string->keyword "pow")) '(2 8) empty) ; 256
1
2
3
4
5
6
7
(define (f #:base x #:pow y)
  (expt x y))
(f #:base 2 #:pow 8) ; 256
(keyword-apply f '(#:base #:pow) '(2 8) empty) ; 256
(keyword-apply f (list
                  (string->keyword "base")
                  (string->keyword "pow")) '(2 8) empty) ; 256


For more about the use of key­word argu­ments, see func­tions.

Path strings



A path string is not a dis­tinct type, but rather the sub­set of strings that can be con­verted into a valid path. Many Racket func­tions that manip­u­late paths will accept either a path or a path string, so it’s often not nec­es­sary to man­u­ally con­vert the input argu­ment with string->path:


(file-exists? (string->path "/usr/bin/racket")) ; #t
(file-exists? "/usr/bin/racket") ; #t
1
2
(file-exists? (string->path "/usr/bin/racket")) ; #t
(file-exists? "/usr/bin/racket") ; #t


Paths



A path rep­re­sents a file loca­tion. Paths are sim­i­lar to strings. But it’s a tidge mis­lead­ing to cat­e­go­rize them as “string­like”, because they’re made of a sequence of bytes rather than Uni­code char­ac­ters. In Racket, this is called a byte string. It’s a dis­tinct data type from an ordi­nary Uni­code string.



In prac­tice, many paths can be con­verted to an ordi­nary string and back to a path. So as a con­ve­nience, Racket offers path->string and string->path:


"/usr/bin/racket"
(string->path "/usr/bin/racket")
(path->string (string->path "/usr/bin/racket"))
1
2
3
"/usr/bin/racket"
(string->path "/usr/bin/racket")
(path->string (string->path "/usr/bin/racket"))

"/usr/bin/racket"
#<path:/usr/bin/racket>
"/usr/bin/racket"
1
2
3
"/usr/bin/racket"
#<path:/usr/bin/racket>
"/usr/bin/racket"


But some paths can’t be con­verted to a string with­out los­ing data. In gen­eral, path->string is intended for mak­ing user-read­able ver­sions of paths. It’s unwise to manip­u­late paths by con­vert­ing them to strings and back again. Bet­ter to use Racket’s ded­i­cated path-manip­u­la­tion func­tions.
