Making decisions under pressure

by Gregory Brown
Imagine that you're working on an application that help peoples find nearby yard sales.

The home page for your site is a giant map with pins identifying each yard sale, and it uses a combination of IP and browser based geolocation to automatically select a location to display.

The code that's being used for geolocation just produces a lat/long coordinate, and so to fill out the search field on the home page with a useful place name, it needs to be reverse geocoded. This process is currently being done server-side, and we'll soon see how that can lead to big headaches.

The reverse-geocoding API is fairly slow to respond, and a few weeks ago as your site started to get more traffic, requests started queueing up leading to a very slow load time for the homepage. So you did what any good engineer would do: You doubled the number of server instances to field more requests simultaneously.

That worked, for a little while. Then today, some super famous incredibly important yard salin' newsletter linked to your site, and traffic spiked like crazy. You again, did what the responsible engineer might do and doubled the number of server instances again. For about ten minutes, this seemed to stabilize things.

Then the alert emails starting coming in. The home page is responding with a 500 error! The exception reports are showing that the library you're using for the reverse geocoding API is raising an exception. This means that all visitors are seeing a generic "We're sorry, something went wrong" page instead of enjoying your amazing website.

Those emails keep coming and coming. Now you have two problems: A broken home page, and a soon-to-be-broken and clearly misconfigured exception monitoring system.

It's time to put this fire out, and your smoke detector is also on fire. So where do you begin? Are there questions you need answered first? Or are there simple actions you can take that will end the state of emergency so that you can think properly?
