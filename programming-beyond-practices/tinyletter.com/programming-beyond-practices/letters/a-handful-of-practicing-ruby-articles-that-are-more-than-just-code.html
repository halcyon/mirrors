A handful of Practicing Ruby articles that are "more than just code"

by Gregory Brown
Hello Developer Friend!

I started this newsletter to explore ideas for Programming Beyond Practices out in the open. By doing that, I'll discover what's interesting and useful to you, and your feedback will help me make a better book. 

I've been thinking about these "beyond the code" topics for years... but the current book idea is brand new. Like nearly every other big, ambitious project I've done, this one stems from a gut feeling that is pretty much only an hour old at this point. PrawnPDF was born this way, Mendicant University was born this way, and so was Practicing Ruby. I can't tell you exactly what lies on the path ahead, because I don't know that yet, and I will be relying on your help to figure out the path we end up taking.

For now, let's glimpse into the past a bit. If you squint a little, you'll be able to get an idea for how this future book will look by reading the following Practicing Ruby articles:
  • Sustainable maintenance: A collaboration w. Eric Hodel where we talk about letting external changes drive internal code quality improvements, treating all code without tests as legacy code (ala Michael Feathers), and expanding functionality of reusable code via stable extension points rather than implementing tons and tons features in a single codebase.
     
  • Problem Discovery comes before Problem Solving: A brief case study on how to explore a business domain in the early stages of the requirements discovery phase to look for opportunities to write less code before jumping head first into an overly complicated "perfect solution".
     
  • Guidelines for growing a stable codebase: A collaboration with Jordan Byron where we discuss how to keep a project stable as it grows, even if you don't have a huge maintenance budget to work with. The key ingredients include: Incremental feature development, outside-in reviews of work in progress, active system monitoring at both the technical and business level, an easy way to revert changes, a clearly defined plan for dealing with any defects that are discovered in production, and the cultivation of a comprehensive regression testing safety net.
     
  • From requirements discovery to release: The steps involved in going from a blank text file and an idea in your head to a (tiny) prototype that can be shared with others for their feedback, as seen in the context of a somewhat contrived little game I built.
Practicing Ruby is a code-heavy publication, so these articles were mostly written with that audience in mind. I imagine the Programming Beyond Practices book will have less code, but hopefully retain some degree of rigor through rich examples and case studies. I still need to imagine what that will look like, but for now think of something similar in spirit to the articles above, but without the need to dig through source code.

This is just a starting point... a tiny kernel of an idea that needs much, much more time to percolate. But hopefully it at least captures your interest! I have many more stories to share, so if you have specific topics you are interested in, please reply to this email and I'll see if I can dig up something related to what you're struggling with.

In any case, thanks for going on a weird and wild ride with me. It's gonna be fuuuuuuun! (and hard, and scary, and messy... but mostly fun)
