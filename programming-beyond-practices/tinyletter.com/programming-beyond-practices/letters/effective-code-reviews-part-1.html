Effective Code Reviews, Part 1

by Gregory Brown
When done well, code reviews will help you produce better software.

That said, it's only possible to give an effective review if you know what to look for.

For the most part, reviews that focus primarily on coding style, structure, and idioms tend to produce mediocre results at best. They may in some ways lead to better code, but better code on its own does not mean better software.

As an analogy, it's easy to fall into a trap where code reviews become nothing more than an exercise in technical proofreading. In doing this, you end up focusing on computational equivalent of grammar and punctuation and the particular wording of particular phrases, and you forget that the main goal is to tell an meaningful story. When you cannot see the forest for the trees, bad things happen.

How can we do better than that? It's a tricky question without a "one size fits all" answer.

To help you identify the kinds of questions that will lead you to focus on substance rather than style in code reviews, I'll rattle off a few now and see if you can pick up a pattern.

Got your thinking cap on yet? Good. Let's do this.
 
  • Does this code actually solve the problem it's meant to solve? Check the bug report or feature request, and any supporting documentation or discussions to make sure.
     
  • When shipped, will this improvement increase the value of the project as a whole? Sometimes things sound better on paper than they turn out in practice, and so there's no time like the present to go back to the drawing board if what was built might end up being more trouble than its worth to ship and maintain.

If the answer to either of the above questions is "no", then stop right here. Don't even look at the code. If you're not the one in charge of deciding what features and fixes get worked on and when, get in touch with the person who is, and talk things through. Reviewing code that won't survive in production is a giant waste of time.

OK, the code being reviewed solves a real problem, and the benefits of solving that problem in this particular way outweigh the costs? Great, we can move on.
 
  • Is the code able to run somewhere other than the original developer's machine? Running live in production as a developers-only feature is best, but a staging environment, or even another developer's machine can be a good enough smoke test. If you're not able to easily verify this, your process is broken, and that needs fixing before you can do effective code reviews.
     
  • Did you need to ask someone for help in order to get the code running in order to do your review (whether it's deployed or on your own machine?). If so, either documentation is needed, or whatever setup process you did needs to be automated, or both.
     
  • Now actually try out the planned improvement from the perspective of the people who use your software (or from the perspective of a developer, if it's an internal change). Did the fix or feature work the way you'd expect it to? Was anything awkward, missing, or broken? Have a conversation about that, then pause the review until those issues are dealt with. If this is an early stage review, you can choose to focus on certain issues and ignore other details that you know will be sorted out later, but it pays to be explicit here.

If you've made it this far, you now have working software that solves a real, worthwhile problem, and it's running somewhere other than a single developer's machine. It's not crashing and burning in weird ways, and it's not missing any obviously important bits of functionality. You know that anyone else who might want to review this work will be able to do so easily, because the process of getting the code up and running has either been documented or automated, or both.

How much code have you read at this point in the review? In a well organized project, the answer may be "none". But you've done quite a bit to uphold high software quality standards, just by looking at things from the outside in. It may have only costed you a few minutes of time, but it's a critically important first step to giving a great code review.
